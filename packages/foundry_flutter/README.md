# foundry_flutter

Flutter integration layer for the [Foundry MVVM framework](../foundry_core). Provides the widgets, scope wiring, and state management plumbing that connects your `FoundryViewModel` / `@FoundryView` annotated classes to the Flutter widget tree.

This package re-exports everything from `foundry_core` and adds Flutter-specific widgets on top. In most Flutter apps you only need to import `foundry_flutter` — not `foundry_core` directly.

Part of the Foundry framework - a compile-time MVVM architecture for Flutter.

- [foundry_core](https://pub.dev/packages/foundry_core) - DI, scopes, lifecycles
- [foundry_flutter](https://pub.dev/packages/foundry_flutter) - View + ViewModel binding
- [foundry_navigation_flutter](https://pub.dev/packages/foundry_navigation_flutter) - Typed navigation runtime
- [foundry_annotations](https://pub.dev/packages/foundry_annotations) - Annotations for code generation
- [foundry_generator](https://pub.dev/packages/foundry_generator) - Code generation and validation

## Concepts

| Concept | Class | Role |
|---|---|---|
| View base class | `FoundryView<TViewModel, TState>` | Extend this instead of `StatefulWidget`. Handles ViewModel lifecycle automatically. |
| DI scope widget | `FoundryScope` | Injects a `Scope` into the widget tree via `InheritedWidget`. |
| State-driven builder | `FoundryBuilder<S>` | Rebuilds its subtree every time a `StateEmitter` emits. |
| Side-effect listener | `FoundryListener<S>` | Runs a callback on every emission without rebuilding. |
| Selective builder | `FoundrySelectorBuilder<S>` | Rebuilds only when your selector function returns `true`. |
| State mixin | `FoundryState` | The underlying `State` implementation — exposed for custom views. Provides `isBound`, `viewModel`, `currentState`, and `oldState` accessors for subclasses that need direct ViewModel access during the build phase. |

## Setup

```yaml
dependencies:
  foundry_flutter: ^0.0.1
```

Your app entry point needs to wrap its widget tree with an `FoundryScope` that provides the root DI `Scope`:

```dart
import 'package:foundry_flutter/foundry_flutter.dart';

void main() {
  final scope = GlobalScope.create();
  registerGeneratedGraph(scope); // generated by foundry_generator
  runApp(
    FoundryScope(
      scope: scope,
      child: const MyApp(),
    ),
  );
}
```

## `FoundryView`

Extend `FoundryView<TViewModel, TState>` instead of `StatefulWidget`. The framework creates a child scope per view instance, resolves `TViewModel` from that view scope, reads its current state, calls `onInit`, subscribes to its state stream, and calls `buildWithState` on every emission.

```dart
// home_view.dart
part 'home_view.g.dart'; // generated by foundry_generator

@FoundryView(route: '/home')
class HomeView extends FoundryView<HomeViewModel, HomeState> {
  const HomeView({super.key});

  @override
  Widget buildWithState(BuildContext context, HomeState? oldState, HomeState state) {
    if (state.isLoading) {
      return const CircularProgressIndicator();
    }
    return ListView(
      children: state.items.map((item) => Text(item)).toList(),
    );
  }
}
```

> The `@FoundryView` annotation and the generated `HomeViewRoute` / `pushHomeView()` helpers come from [`foundry_generator`](../foundry_generator). `foundry_flutter` only needs the `FoundryView` base class.

> Important: because `FoundryView` reads `viewModel.state` before `onInit()` runs, your ViewModel must already have an initial state by the time it is resolved from the scope.

> Inside `buildWithState`, `FoundryScope.of(context)` resolves the same per-view scope used to bind the view. This means any dependency lookup from that context sees the view-scoped `TViewModel` instance.

> Register ViewModels with `Lifetime.scoped` so each pushed view gets its own instance. Keep repositories and long-lived models as singletons.

### `oldState`

`buildWithState` receives both `oldState` (nullable — `null` on first build) and the new `state`. Use `oldState` to drive animations or one-off UI transitions:

```dart
@override
Widget buildWithState(BuildContext context, HomeState? oldState, HomeState state) {
  final wasEmpty = oldState?.items.isEmpty ?? true;
  final isNowPopulated = state.items.isNotEmpty;

  return AnimatedSwitcher(
    duration: const Duration(milliseconds: 300),
    child: isNowPopulated
        ? ItemList(items: state.items)
        : const EmptyPlaceholder(),
  );
}
```

## `FoundryScope`

`FoundryScope` is an `InheritedWidget` that provides a `Scope` (the DI container) to all descendants. `FoundryView` resolves its `TViewModel` from the nearest `FoundryScope` automatically.

```dart
// Root scope at app level:
FoundryScope(
  scope: myGlobalScope,
  child: const MyApp(),
)
```

If no `FoundryScope` is found above the given `context`, `FoundryScope.of` throws a `StateError`. This is by design — every widget that uses the DI container must be within a `FoundryScope`.

### Child scopes

`FoundryScope.childScope()` creates an automatically managed child scope that is disposed when the subtree is removed. Use this to give a feature sub-tree its own scoped registrations:

```dart
FoundryScope.childScope(
  child: const CheckoutFeature(),
)
```

The child scope is created as a child of the nearest ancestor `FoundryScope`. It inherits all parent registrations and can add or shadow its own. The scope is disposed automatically when the widget is removed from the tree, and is recreated if the parent scope instance changes.

## `FoundryBuilder`

Rebuilds its subtree whenever the bound `StateEmitter` (a `ViewModel` or `StatefulService`) emits a new state. Equivalent to a `StreamBuilder` but without the need to handle `AsyncSnapshot`.

```dart
FoundryBuilder<CartState>(
  emitter: FoundryScope.of(context).resolve<CartModel>(),
  builder: (context, oldState, state) {
    return Text('${state.items.length} items in cart');
  },
)
```

Both `oldState` and `state` are always non-null inside `builder` (the first call receives the current state from `emitter.state`).

---

## `FoundryListener`

Runs a side-effect callback on every state emission without triggering a widget rebuild. The `child` is rendered as-is.

```dart
FoundryListener<HomeState>(
  emitter: FoundryScope.of(context).resolve<HomeViewModel>(),
  listener: (context, oldState, state) {
    if (state.error != null && oldState?.error != state.error) {
      ScaffoldMessenger.of(context).showSnackBar(
        SnackBar(content: Text(state.error!)),
      );
    }
  },
  child: const HomeContent(),
)
```

> **Important:** The `listener` callback is NOT called for the initial state. It fires only for emissions that occur after the widget is mounted. When it does fire, `oldState` is always the previous known state (never `null`). If you also need to react to the current state on first mount, read `emitter.state` directly in your build method or combine with `FoundryBuilder`.

---

## `FoundrySelectorBuilder`

Like `FoundryBuilder`, but rebuilds only when the `selector` function returns `true`. Use this to prevent unnecessary rebuilds in deeply nested subtrees.

```dart
// Only rebuilds when the loading flag changes.
FoundrySelectorBuilder<HomeState>(
  emitter: FoundryScope.of(context).resolve<HomeViewModel>(),
  selector: (old, next) => old?.isLoading != next.isLoading,
  builder: (context, state) {
    return state.isLoading
        ? const LoadingSpinner()
        : const HomeBody();
  },
)
```

A few behavioral notes:

- The `selector` is **not called on the initial build**. The widget always renders once immediately with the current state from `emitter.state`.
- The `builder` callback receives `(BuildContext context, S state)` — there is no `oldState` parameter, unlike `FoundryBuilder`. If you need `oldState` to drive your UI, use `FoundryBuilder` instead.
- Even when `selector` returns `false` and no rebuild occurs, the widget keeps tracking the latest state. The next time `selector` returns `true` it will use the most recent state — not a stale snapshot.

## Putting it all together

A typical feature screen combining several of these widgets:

```dart
@FoundryView(route: '/home')
class HomeView extends FoundryView<HomeViewModel, HomeState> {
  const HomeView({super.key});

  @override
  Widget buildWithState(BuildContext context, HomeState? oldState, HomeState state) {
    final HomeViewModel viewModel = FoundryScope.of(
      context,
    ).resolve<HomeViewModel>();

    return FoundryListener<HomeState>(
      emitter: viewModel,
      listener: (ctx, _, next) {
        if (next.error != null) {
          ScaffoldMessenger.of(ctx)
              .showSnackBar(SnackBar(content: Text(next.error!)));
        }
      },
      child: Scaffold(
        appBar: AppBar(title: const Text('Home')),
        body: FoundrySelectorBuilder<HomeState>(
          emitter: viewModel,
          selector: (old, next) => old?.items != next.items,
          builder: (_, s) => ItemList(items: s.items),
        ),
        floatingActionButton: FoundryBuilder<HomeState>(
          emitter: viewModel,
          builder: (_, __, s) => s.isLoading
              ? const CircularProgressIndicator()
              : FloatingActionButton(
                  onPressed: viewModel.refresh,
                  child: const Icon(Icons.refresh),
                ),
        ),
      ),
    );
  }
}
```

## ViewModel lifecycle

`FoundryState` (the `State` backing every `FoundryView`) manages the full ViewModel lifecycle automatically:

| Event | ViewModel method called |
|---|---|
| Widget added to tree | `onInit()` |
| App foregrounded (`resumed`) | `onResumed()` |
| App backgrounded (`inactive`, `hidden`, `paused`) | `onPaused()` |
| Widget removed from tree | `onDispose()` |
| Unhandled error in state stream or during binding | `onError(error, stackTrace)` |

You never call these methods yourself — override them in your `FoundryViewModel` subclass. Initialize the first state before `onInit()` if the ViewModel will be bound by `FoundryView`.

```dart
@FoundryViewModel()
class HomeViewModel extends FoundryViewModel<HomeState>
    with _$HomeViewModelHelpers {

  final ItemRepository _repo;

  HomeViewModel(this._repo) {
    emitNewState(const HomeState(isLoading: true));
  }

  @override
  Future<void> onInit() async {
    await safeAsync(() async {
      final items = await _repo.fetchAll();
      emitNewState(HomeState(items: items));
    });
  }

  Future<void> refresh() => safeAsync(() async {
    emitNewState(state.copyWith(isLoading: true));
    final items = await _repo.fetchAll();
    emitNewState(state.copyWith(isLoading: false, items: items));
  });
}
```

> `safeAsync`, `setError`, and `clearError` are generated by [`foundry_generator`](../foundry_generator) when you annotate your ViewModel with `@FoundryViewModel()`.

When you need the current view's ViewModel instance inside `buildWithState`,
resolve it from the scope bound to that view:

```dart
final HomeViewModel viewModel = FoundryScope.of(context).resolve<HomeViewModel>();
```

### `FoundryState` accessors for custom views

If you need finer control over the view lifecycle — for example to add custom bindings or hook into build logic — extend `FoundryState` directly rather than `FoundryView`:

```dart
class MyCustomView extends FoundryView<MyViewModel, MyState> {
  const MyCustomView({super.key});

  @override
  State<MyCustomView> createState() => _MyCustomViewState();
}

class _MyCustomViewState
    extends FoundryState<MyCustomView, MyViewModel, MyState> {
  @override
  Widget buildWithState(BuildContext context, MyState? oldState, MyState state) {
    // isBound — true once the ViewModel has been resolved from the scope
    // viewModel — the resolved TViewModel (throws if not yet bound)
    // currentState — the latest state (throws if no state emitted yet)
    // oldState — the previous state (null on first build)
    return Text(state.toString());
  }
}
```

These accessors are particularly useful in subclasses that add animation controllers or other widget-tree state that must coordinate with the ViewModel lifecycle.
